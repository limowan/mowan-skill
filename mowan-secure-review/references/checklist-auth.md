# 认证与授权安全清单

认证和授权是大多数应用的第一道防线。绕过认证 = 直接进入系统。

---

## 密码安全

### 检查项

1. **密码存储算法**
   - Grep 密码哈希相关代码
   - 安全：bcrypt、argon2、scrypt（带 salt）
   - 不安全：MD5、SHA1、SHA256（无 salt 或固定 salt）、明文存储
   - 检查 cost/rounds 参数是否合理（bcrypt >= 10）

2. **密码强度策略**
   - 是否有最小长度要求（建议 >= 8）
   - 是否检查常见弱密码（top 1000 密码黑名单）
   - 是否允许纯数字密码

3. **密码传输**
   - 登录接口是否走 HTTPS
   - 密码是否在前端做了不必要的处理（如前端 MD5 后传输 — 这不是安全措施）

---

## 会话管理

### 检查项

1. **会话固定 (Session Fixation)**
   - 登录成功后是否重新生成 Session ID
   - 是否接受 URL 中的 Session ID

2. **会话劫持**
   - Session Cookie 是否设置了 `HttpOnly`
   - Session Cookie 是否设置了 `Secure`（HTTPS 环境）
   - Session Cookie 是否设置了 `SameSite`

3. **会话超时**
   - 是否有空闲超时（idle timeout）
   - 是否有绝对超时（absolute timeout）
   - 登出时是否在服务端销毁会话（而非仅删除客户端 Cookie）

4. **并发会话**
   - 是否限制同一账号的并发会话数
   - 密码修改后是否使其他会话失效

## JWT 实现

### 检查项

1. **算法混淆攻击**
   - 是否在验证时指定了允许的算法（白名单）
   - 是否可能被 `alg: none` 绕过
   - 是否可能被 RS256 → HS256 混淆攻击（用公钥作为 HMAC 密钥）

2. **密钥强度**
   - HMAC 密钥是否足够长（>= 256 bit）
   - 密钥是否硬编码在代码中（→ 转到 checklist-secrets.md）
   - 密钥是否使用了安全随机数生成

3. **Token 内容**
   - Payload 中是否包含敏感信息（密码、完整个人信息）
   - 是否设置了合理的过期时间（`exp`）
   - 是否使用了 `iat`（签发时间）和 `nbf`（生效时间）

4. **Token 生命周期**
   - 是否有 Token 刷新机制
   - 登出时是否有 Token 黑名单/撤销机制
   - 密码修改后旧 Token 是否失效

5. **Token 存储（前端）**
   - 存储在 HttpOnly Cookie → 安全（防 XSS 读取）
   - 存储在 localStorage → 有 XSS 风险
   - 存储在内存 → 安全但刷新丢失

## OAuth / OIDC 实现

### 检查项

1. **Redirect URI 验证**
   - 是否严格校验 redirect_uri（完全匹配，非前缀匹配）
   - 是否允许开放重定向（`redirect_uri=https://evil.com`）
   - 是否允许 localhost（生产环境不应允许）

2. **State 参数**
   - 是否使用了 state 参数防 CSRF
   - state 是否为不可预测的随机值
   - 回调时是否验证了 state

3. **Token 处理**
   - Access Token 是否在前端暴露（隐式流的风险）
   - Refresh Token 是否安全存储
   - 是否使用了 PKCE（推荐用于公开客户端）

## 密码重置

### 检查项

1. **重置令牌安全**
   - 令牌是否使用安全随机数生成（>= 128 bit）
   - 令牌是否有过期时间（建议 <= 1 小时）
   - 令牌使用后是否立即失效（一次性）
   - 新的重置请求是否使旧令牌失效

2. **信息泄露**
   - 重置请求是否泄露账号是否存在（"该邮箱未注册" vs 统一提示）
   - 重置邮件是否泄露了用户名或其他信息

3. **流程绕过**
   - 是否可以直接访问重置密码页面而不需要有效令牌
   - 令牌是否在 URL 中（可能被 Referer 泄露）
   - 是否可以通过修改用户 ID 参数重置他人密码

## 多因素认证 (MFA/2FA)

### 检查项

1. **TOTP 实现**
   - 密钥是否加密存储
   - 是否允许时间窗口容差（通常 ±1 个周期）
   - 是否防止重放（同一个 OTP 码不能用两次）

2. **绕过风险**
   - 是否有"记住设备"功能，其 token 是否安全
   - 备用码是否安全生成和存储
   - 2FA 绑定流程本身是否需要验证身份

3. **降级攻击**
   - 是否可以通过 API 直接绕过 2FA 步骤
   - 密码重置是否会绕过 2FA

---

## 速率限制

### 检查项

1. **登录接口**
   - 是否有失败次数限制
   - 限制是基于 IP、账号、还是两者结合
   - 锁定后的解锁机制（时间自动解锁 vs 需要人工）

2. **绕过方式**
   - 是否可以通过修改 `X-Forwarded-For` 绕过 IP 限制
   - 是否可以通过大小写变化绕过账号限制（`admin` vs `Admin`）
   - 是否可以通过添加空格/特殊字符绕过

3. **其他敏感接口**
   - 验证码发送接口是否有限流（防短信轰炸）
   - 密码重置接口是否有限流
   - API Key 生成接口是否有限流

---

## 账号枚举

### 检查项

1. **注册接口**
   - "该邮箱已注册" → 泄露了用户是否存在
   - 建议：统一返回"如果该邮箱已注册，您将收到邮件"

2. **登录接口**
   - "用户名不存在" vs "密码错误" → 泄露了用户是否存在
   - 建议：统一返回"用户名或密码错误"

3. **时序攻击**
   - 用户存在时（需要验证密码）和不存在时（直接返回）的响应时间差异
   - 建议：无论用户是否存在，都执行一次密码哈希比较

4. **密码重置**
   - 是否通过不同响应泄露邮箱是否注册
