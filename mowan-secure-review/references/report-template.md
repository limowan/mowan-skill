# 安全审计报告模板

生成报告时按此模板组织内容。每个漏洞发现必须包含完整的攻击路径描述。

---

## 报告结构

```markdown
# 安全审计报告

**项目**: {project_name}
**审计日期**: {date}
**审计范围**: {scope — 全量代码审计 / 指定模块审计}
**技术栈**: {detected_stack_summary}
**审计方法**: 静态代码分析（攻击者视角）

---

| 严重程度 | 数量 |
|---------|------|
| Critical | {n} |
| High | {n} |
| Medium | {n} |
| Low | {n} |
| Info | {n} |

**一句话总结**: {用一句话概括项目的安全状况和最大风险点}

**最紧急的修复**: {列出需要立即修复的 1-3 个问题}

---

## 漏洞详情

|------|-----|

| **攻击面** | {哪个入口点 / 哪个接口} |
| **影响** | {攻击者利用成功后能做什么} |
| **置信度** | {高 / 中 / 低 — 基于静态分析的确定程度} |

{这个漏洞是什么，为什么存在}

{从攻击者视角，一步步描述如何利用这个漏洞}

**相关代码**:

（按严重程度从高到低排列所有发现）

---
```

## 报告撰写原则

1. **每个发现必须有攻击路径**：不是"这里可能有问题"，而是"攻击者可以这样做"
2. **修复建议必须具体**：给出代码级别的修复方案，不是"加强安全防护"
3. **置信度要诚实**：静态分析无法确认的，标注为"需运行时验证"
4. **不重复报告**：同一类问题在多处出现时，报告一次并列出所有位置
5. **按严重程度排序**：读者最先看到最紧急的问题
6. **攻击链单独成节**：组合漏洞的影响往往大于单个漏洞之和
