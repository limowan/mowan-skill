# 严重程度定义

## 五级分类

### Critical（严重）

远程代码执行、全量数据泄露、全局认证绕过、支付金额篡改。

攻击者可以在无需特殊前提条件下，对系统造成不可逆的重大损害。

典型场景：
- 未过滤的用户输入直接进入 SQL 查询（登录接口）
- 反序列化漏洞导致 RCE
- 硬编码的管理员凭证
- JWT Secret 泄露在公开代码中
- 支付回调未验签，可伪造支付成功

### High（高危）

显著的数据泄露、单用户账户接管、权限提升、敏感操作的 CSRF。

攻击者可以获取大量敏感数据或控制其他用户的账户，但影响范围或利用条件有一定限制。

典型场景：
- 存储型 XSS（管理后台）
- IDOR 导致任意用户数据访问
- 密码重置流程可被劫持
- API Key 在前端代码中硬编码
- 环境变量通过错误页面泄露（含数据库连接串、密钥等）

### Medium（中危）

有限的数据泄露、需要用户交互才能利用、特定条件下的安全绕过。

攻击者需要特定条件（用户点击链接、特定配置、特定时序）才能利用。

典型场景：
- 反射型 XSS（需要用户点击恶意链接）
- 登录接口无速率限制（可暴力破解）
- 详细的错误信息泄露堆栈和内部路径
- CORS 配置过于宽松
- 环境变量中的非关键配置泄露

### Low（低危）

轻微的信息泄露、最佳实践违反但有实际安全影响。

攻击者获得的信息有限，且难以直接利用造成实质损害。

典型场景：
- 响应头泄露服务器版本
- 非敏感路径的目录列表
- 非敏感页面缺少安全响应头
- Cookie 缺少 Secure 标志（但站点已全站 HTTPS）

### Info（信息）

观察记录、加固建议、当前无直接可利用性但值得关注。

典型场景：
- 过时但无已知漏洞的依赖
- 代码质量问题可能在未来演变为安全问题
- 防御纵深建议（增加额外防护层）
- 日志中记录了非必要的信息（但未包含敏感数据）

## 定级原则

1. **按实际影响定级**，不按理论最大影响
2. **考虑现有防御**：如果有其他防御层有效阻止了利用，降级
3. **考虑利用条件**：需要管理员权限才能触发的漏洞，比公开接口的同类漏洞低一级
4. **攻击链加成**：多个低级别发现可组合利用时，组合后的严重程度可能高于单个发现
5. **业务上下文**：支付相关的 Medium 级漏洞可能比非支付场景的 High 级漏洞更紧急
